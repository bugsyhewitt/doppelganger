## Cut line (one sentence)

doppelganger v0.1 reliably **detects and DIFFERENTIALLY CONFIRMS the HTTP/1.1 desync family**, with first-class pipelining-false-positive discrimination and safe-testing defaults, emitting suite findings — i.e. smuggler.py's coverage **plus the confirmation/correctness layer it never had**, headless where PortSwigger's HTTP Request Smuggler is Burp-locked.

Rationale for the line: v0.1 must *decisively beat smuggler.py* on HTTP/1.1 — which means not just detection but a **confirmation stage + pipelining discrimination** (the correctness gap that makes smuggler.py untrustworthy against real targets). HTTP/2 and the parser-discrepancy engine are the long-term differentiators vs. Burp, but each is a separate sub-project (a whole low-level H2 stack, or the full V-H/H-V engine) that would blow a 500K single-Worker budget if bundled. Ship the reliable HTTP/1.1 core first.

## v0.1 acceptance criteria (testable)

1. **HTTP/1.1 desync detection family** — CL.TE, TE.CL, TE.TE (transfer-encoding obfuscation dictionary), CL.0, and conflicting/duplicate Content-Length. One shared raw-HTTP/1.1 probe engine.
2. **Two-stage engine: timing-based detection → differential-response confirmation.** Emit a **candidate** on a significant timing delta (back-end made to wait for bytes that never arrive); upgrade to **confirmed** when a smuggled prefix makes a follow-up request return a materially different response. Report the two states distinctly.
3. **Pipelining-vs-smuggling false-positive discrimination (headline correctness feature).** Explicit connection-reuse control; any effect that reproduces **only** under client-side connection reuse is flagged/suppressed as probable pipelining (a client-side artifact), **not** reported as a server-side desync. This is the single most important correctness feature — the "false false-positive" that discredits naive tools.
4. **Safe-testing defaults.** Test CL.TE before TE.CL (a TE.CL timing probe can hang and disrupt *other* users if the target is actually CL.TE); per-probe connection isolation; bounded + randomized timeouts; a `--safe`/production mode; never leave a poisoned socket in a shared pool.
5. **Raw-request transport (mandatory architecture).** Smuggling probes go through a **dedicated raw-socket sender** with byte-exact CL/TE control and **no header normalization**. The suite HTTP client is used **only** for baseline/differential *well-formed* requests. A normalizing high-level client **cannot** carry the probe — design the shared-client integration around this split from day one.
6. **Output.** Every candidate/confirmed desync emitted in the suite finding schema (technique, discrepancy type `X.Y`, evidence bytes, timing delta, confirmation status, reproduction payload) + SARIF 2.1.0 + HackerOne markdown via `h1-reporter`.
7. **CI test lab** (see Testability) + the suite `ship_gate` wheel test.

## Explicitly NOT in v0.1 (deferred)

- **All HTTP/2** (H2.CL, H2.TE, H2 request tunnelling/splitting, downgrade desync) → **v0.2, the highest-value follow-up.** Requires a separate low-level H2 stack (`h2spacex` / scapy / raw sockets + ALPN) that **cannot reuse** the HTTP/1.1 raw sender — a distinct engine. (Note: high-level libs `h2`/`httpx` *validate on send* and refuse the prohibited headers these attacks need.)
- **H2C cleartext-upgrade smuggling** — different attack class (edge-proxy bypass); bolt on after H2 lands.
- **Client-side / browser-powered desync (CSD)** — weaponization needs a victim browser; out of scope for a headless CLI. (The underlying CL.0 server bug is already detected by criterion 1.)
- **0.CL, double-desync, Expect-based desync, early-response gadgets** (2025 "HTTP/1.1 Must Die") → v0.3; needs the CL.0 primitive (in v0.1) as a prerequisite plus gadget discovery.
- **Parser-discrepancy V-H/H-V engine** (HRS v3 flagship) — a research-grade sub-project on its own.
- **Connection-state / first-request routing probes** → v0.2 (cheap once connection control exists, but not core).
- **Full weaponization** (cache poisoning, request capture, PoC chaining) — doppelganger is a detection/confirmation tool; keep it there for scope discipline.

## Testability (CI-friendly lab)

A desync needs a **discrepant pair** — a front-end and back-end that parse message length differently. Pin old versions so the discrepancy is deterministic.

- **Integration (gated marker) — docker-compose discrepant pair.** e.g. HAProxy 1.7.9 (CVE-2019-18277) + gunicorn 20.0.4, or the ZeddYu HAProxy→ATS→nginx stack (frozen versions = reproducible). Reliably reproduces a specific `X.Y` desync.
- **Unit (fast, deterministic) — in-process raw-socket mock pair.** A tiny `asyncio`/raw-socket "front-end" and "back-end" with **opposite length rules** and controllable hang/timeout, synthesizing any `X.Y` discrepancy on demand with millisecond-stable timing. **This is what makes the timing-detection, differential-confirmation, and pipelining-discrimination logic actually unit-testable** without container flakiness.
- **Exclude** PortSwigger Web Security Academy labs (cloud-hosted, per-user, authenticated, un-scriptable).

## Safety

- **R5 (untrusted input):** response bytes are **data, never instructions**.
- **Scope enforcement before ANY probe** via **`scan-primitives`** (baseline/differential requests); the raw-socket probe path must itself honor the same `Scope` object.
- Safe-testing defaults are a v0.1 *acceptance criterion* (criterion 4), not an optional nicety — this tool can disrupt shared back-ends if careless.

## Shared infra (resolved — Option C, 2026-07-02)

- **HTTP client:** use **`scan-primitives`** for baseline/differential *well-formed* requests only. doppelganger's desync probes are **raw by necessity** (byte-exact CL/TE via a dedicated raw-socket sender) — the shared client cannot carry them. Pass the `Scope` object into the raw sender so probes are scope-checked too.
- **Findings:** per-tool `Finding` to the **pinned contract** in `scan-primitives/SPEC.md` (`cwe_id=444`); SARIF 2.1.0; adopt **`h1-reporter`**.
- Build **`scan-primitives`** before this tool.
